## 一、项目背景与定位

### 1.1 问题
微信小程序因ICP许可证问题无法上线，导致平台无法正常运营。已确认合作意向但未开展任何业务，当前仅通过微信群与上下游沟通。

### 1.2 解决方案定位
**内部工具 ≠ 替代平台，而是桥梁**

- 将平台核心功能搬到内部Web工具上
- 让试点业务先跑起来，积累真实运营数据
- 不影响未来小程序上线后的迁移
- 小程序上线后，内部工具可平滑过渡或退役

### 1.3 试点规模
| 角色 | 人数 | 说明 |
|---|---|---|
| 用户端 | ≤30人 | 合作社、农产品企业等有吊运需求的企业用户 |
| 飞手端 | ≤50人 | 无人机操作员 |

## 二、开发方案选型

### 2.1 方案对比

| 方案 | ICP要求 | 开发周期 | 用户体验 | 迁移成本 | 推荐度 |
|---|---|---|---|---|---|
| **H5 Web App（推荐）** | 内部使用无需ICP | 2-3周 | 手机浏览器，良好 | 低（逻辑可复用） | ⭐⭐⭐⭐⭐ |
| 微信小程序测试版 | 无需ICP | 2-3周 | 原生体验 | 零（已是最终形态） | ⭐⭐⭐⭐ |
| 企业微信应用 | 无需ICP | 3-4周 | 企业微信内 | 中 | ⭐⭐⭐ |

### 2.2 推荐方案：移动端优先的H5 Web App

**选择理由：**
1. **无需ICP**：内部工具不面向公众，无需备案
2. **开发最快**：复用现有小程序的业务逻辑和API
3. **访问便捷**：微信内直接打开链接，无需安装
4. **迁移成本低**：核心逻辑与小程序一致，未来可平滑迁移
5. **双端统一**：用户端和飞手端可共用一套代码框架

**⚠️ 关键设计原则：移动端优先**
- 飞手在野外作业，不可能带着电脑
- 两端都必须是**手机H5网页**，不是PC网页
- UI框架选择移动端优先的（如 Vant、NutUI）
- 响应式设计，但以手机屏幕为主

**技术栈：**
- 前端：HTML/CSS/JavaScript + 移动端UI框架（Vant/ NutUI）
- 后端：复用现有Supabase（PostgreSQL + Edge Functions）
- 部署：静态托管（Vercel/Netlify/自有服务器）
- 访问方式：微信内分享链接

## 三、订单字段设计（核心）

### 3.1 用户下单表单

用户发布吊运需求时，必须填写以下字段：

| 字段名 | 类型 | 说明 | 必填 | 示例 |
|---|---|---|---|---|
| **起始地址** | 地图选点+手动输入 | 货物起点 | ✅ | 重庆市XX县XX镇XX村 |
| **目的地址** | 地图选点+手动输入 | 货物终点 | ✅ | 重庆市XX县XX镇XX村 |
| **距离** | 数字输入（km） | 起点到终点的距离 | ✅ | 15.5 |
| **货物描述** | 文本输入 | 什么东西 | ✅ | 新鲜脐橙 500箱 |
| **重量** | 数字输入（kg） | 货物重量 | ✅ | 250 |
| **价值** | 数字输入（元） | 货物价值 | ✅ | 15000 |
| **落差** | 数字输入（m） | 起降点海拔高度差 | ✅ | 350 |
| **时效要求** | 选择框 | 期望完成时间 | ✅ | 2小时内 / 今天内 / 明天内 |
| **联系方式** | 手机号 | 用户电话 | ✅ | 138xxxx |
| **备注** | 文本输入 | 特殊要求 | ❌ | 需要冷链运输 |

### 3.2 飞手端订单展示

飞手接单时，订单列表和详情必须显示以下关键信息：

**订单列表（卡片式）：**
- 📍 起→终（地址简写）
- 📏 距离：XX km
- ⚖️ 重量：XX kg
- 📈 落差：XX m
- ⏰ 时效：XX小时内
- 💰 价值：￥XX

**订单详情页：**
- 完整显示上述所有字段
- 货物描述
- 联系方式
- 下单时间

### 3.3 数据库字段设计

```sql
-- 订单表
CREATE TABLE orders (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id),
  pilot_id UUID REFERENCES users(id),
  
  -- 地址信息
  origin_address TEXT NOT NULL,
  origin_lat DECIMAL,
  origin_lng DECIMAL,
  dest_address TEXT NOT NULL,
  dest_lat DECIMAL,
  dest_lng DECIMAL,
  
  -- 核心业务字段
  distance_km DECIMAL NOT NULL,        -- 距离（km）
  weight_kg DECIMAL NOT NULL,          -- 重量（kg）
  value_yuan DECIMAL NOT NULL,         -- 价值（元）
  altitude_drop_m DECIMAL NOT NULL,    -- 落差（m）
  urgency TEXT NOT NULL,               -- 时效：2h/today/tomorrow
  
  -- 其他信息
  cargo_desc TEXT NOT NULL,            -- 货物描述
  contact_phone TEXT NOT NULL,         -- 联系电话
  remark TEXT,                         -- 备注
  
  -- 状态
  status TEXT DEFAULT 'pending',       -- pending/accepted/executing/completed/cancelled
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

## 七、与现有代码的复用

### 7.1 可直接复用
- **API层**：`utils/request.js` 的请求封装逻辑
- **数据模型**：Supabase表结构和Edge Functions
- **业务逻辑**：下单、接单、状态流转的核心逻辑

- **存储**：从wx.setStorage改为localStorage/sessionStorage
- **地图**：从微信地图改为高德/腾讯地图JS API
- **表单**：重新设计下单表单，增加距离、重量、价值、落差、时效字段

- 微信支付（线下结算）
- 实名认证（内部信任）
- 保险模块（试点砍掉）

## 九、风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 地图API定位不准 | 用户体验差 | 使用高德/腾讯地图，支持手动输入地址 |
| Supabase免费额度不够 | 功能受限 | 监控使用量，必要时升级 |
| 用户不习惯H5 | 接受度低 | 提供详细使用说明，安排培训 |
| 网络不稳定 | 操作失败 | 增加离线缓存和重试机制 |
| 飞手野外作业网络差 | 无法及时接单 | 考虑离线缓存+短信通知 |
